## Material findings fixed

### Layout presets orphaned per-instance runtime and alarms

Applying a preset replaced every block with newly generated IDs. Settings were saved, but runtime entries belonging to the old IDs and their durable Timer/Pomodoro alarms were not removed in the same operation. A block that disappeared from the UI could therefore keep running in the background and later produce a completion side effect.

Preset application now reuses the first existing instance of every retained block type. Stable IDs preserve the retained block's configuration and runtime. Blocks not present in the new preset are removed through the atomic transaction described below.

### Layout Editor and Options deletion were split transactions

Layout Editor previously saved settings first and then deleted removed runtime IDs one by one. Options block deletion similarly removed settings and runtime in two independent operations. A failure or service-worker restart between those calls could leave orphan notes, tasks, clock runtime, or alarms.

All destructive layout paths now send one validated `replace-start-page-settings` message to the service worker:

- Layout Editor save;
- Options General/preset save;
- Options block deletion;
- onboarding preset application.

The obsolete settings-only preset and block-removal helpers were removed so the unsafe path cannot be reused accidentally.

### Settings, runtime, revision, and alarms were not one recoverable layout unit

The service worker now validates and stamps the proposed settings under the shared `data-write` lock, normalizes runtime against the final validated block IDs, removes legacy runtime, advances the data revision, and reconciles the complete durable alarm set as one recoverable transaction.

If storage or alarm reconciliation fails, exact prior settings, runtime, legacy-key presence, revision metadata, and alarms are restored. Incomplete rollback is surfaced as an `AggregateError` instead of being hidden.

### Preset changes could silently remove customized data

Preset selection now detects when removed blocks contain note/task data, active or progressed clocks, customized titles, or non-default block configuration. The user receives a localized destructive-data confirmation. Untouched built-in defaults do not produce a misleading warning during first-run onboarding.

### Runtime could change after the destructive confirmation

The first implementation guarded only the settings snapshot. Another tab could write a note/task or change a clock after the UI evaluated the warning but before the service worker acquired the transaction lock, allowing newly written per-instance data to be removed without a fresh confirmation.

Every replacement message now carries both settings and runtime timestamps. When block IDs are removed, the service worker compares the current runtime timestamp under `data-write`; a changed runtime rejects the replacement without modifying storage or alarms. Non-destructive replacements continue to preserve the latest runtime without unnecessary rejection.

The remaining unused settings-only reset helper was removed so complete reset continues to have one canonical settings/runtime/alarm owner.

## Regression coverage

Round 15 fixtures verify:

- retained preset block types reuse their existing IDs;
- retained custom block configuration survives preset application;
- untouched built-in defaults do not trigger a destructive-data warning;
- removing a customized block does trigger confirmation logic;
- a rejected asynchronous alarm creation restores exact settings/runtime/revision storage and the exact prior alarm set;
- a successful preset replacement removes orphan note runtime;
- retained link paging and a running Timer survive with the same IDs;
- a removed clock loses both runtime and durable alarm;
- stale settings snapshots are rejected without changing storage or alarms;
- runtime changes after destructive confirmation are rejected without changing storage or alarms;
- a retry with the refreshed runtime snapshot succeeds safely;
- oversized, incomplete, or invalid replacement messages are rejected;
- every destructive layout UI delegates to the service worker;
- settings-only destructive helpers cannot return unnoticed.
