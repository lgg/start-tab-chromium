## Material finding fixed

### Persisted collection limits disagreed across code paths

The service-worker message validator already enforced:

- at most 1,000 Start Page block instances;
- at most 1,000 custom themes;
- at most 10,000 tasks in one Local Tasks instance.

However, local settings/runtime normalization and backup migration had no equivalent bounds. A backup containing 1,001 blocks, 1,001 themes, or 10,001 tasks could therefore be accepted and persisted. The UI could render that data, but the next ordinary save or task mutation was rejected by the message validator. The imported blocks/tasks became effectively uneditable. Large malformed collections also forced avoidable normalization, collision, cloning, and rendering work.

The limits now live in one shared platform-limits module and are applied consistently to:

- service-worker message validation;
- direct settings writes;
- programmatic block creation and duplication;
- Layout Editor add/duplicate controls;
- Options add/duplicate controls;
- local settings normalization;
- custom-theme normalization;
- local task normalization;
- current and legacy backup migration.

Oversized backup data is rejected before normalization and before any local storage, DNR, revision, or alarm mutation. It is not silently truncated. Corrupted pre-existing local data is bounded during normalization so the UI remains within the same editable range used by ordinary mutations.

## Regression coverage

Round 18 executable fixtures verify:

- exact 1,000-block and 1,000-theme settings remain accepted;
- 1,001-block settings messages are rejected;
- direct settings writes reject 1,001 blocks or themes before normalization;
- corrupted local settings normalize to the supported editable limits;
- `canAddBlock` stops additions at the same boundary;
- exact 10,000-task runtime messages remain accepted;
- 10,001-task runtime messages are rejected;
- corrupted local runtime normalizes to 10,000 tasks;
- backups with 1,001 blocks, 1,001 themes, or 10,001 tasks fail before mutation;
- backups exactly at every supported boundary still migrate successfully.

Static and generated-output guards prevent the UI, message validator, settings/runtime normalizers, and backup migration from drifting back to different capacities.
