# Post-Round 22 deep audit — destructive build safety and confirmed unblock state

## Scope

Independent review of every production and validation change merged by PR #99. The audit followed the changed code into adjacent build tooling, generated manifests, worker acknowledgements, blocklist transactions, blocked-page countdown behavior, focus statistics, and the Windows CI contract.

## Material findings and fixes

### 1. A malformed custom output path could delete repository or host data

The builder resolved `--outdir` and immediately passed it to recursive `rm`. Commands such as `node build.mjs --outdir=.` or an output path targeting `src`, a repository parent, or a filesystem root could therefore remove data before compilation began.

Fix:

- output resolution is centralized in a pure safety module;
- repository-local outputs must be under a top-level `build*` directory;
- external outputs are accepted only below the operating-system temporary directory;
- repository roots, source folders, repository parents, temp roots, and filesystem roots are rejected before any removal;
- the safety policy remains compatible with the temporary build directories used by executable fixtures.

### 2. Documented placeholder OAuth IDs still passed explicit Google builds

Round 22 made Google selection explicit, but its format-only regular expression accepted values such as `REPLACE_WITH_GOOGLE_OAUTH_CLIENT_ID.apps.googleusercontent.com`. This contradicted deployment and release documentation stating that placeholders fail.

Fix:

- one shared validator now owns client-ID syntax and placeholder rejection;
- the npm preflight, builder, and generated-output validator all use the same policy;
- Google validation occurs before recursive output cleanup, preserving an existing valid build when credentials are missing or invalid.

### 3. Countdown unblock statistics could still overcount under a race

The blocked page checked that a host was active, sent a normal `unblock` message, and interpreted any successful acknowledgement as proof that it removed the site. Another extension context could remove the site between the check and the serialized worker mutation. The blocked page would then record a countdown unblock even though its own operation changed nothing.

Fix:

- `unblockHost` now returns a boolean from inside the serialized blocklist mutation queue;
- a missing site returns `false` without rewriting storage, DNR rules, or the data revision;
- the service worker includes the actual mutation result in its acknowledgement;
- the blocked page records statistics only when `changed === true`.

## Regression coverage

Round 23 executable fixtures verify:

- two concurrent unblock attempts report exactly one `true` result;
- stale repeated unblocks do not rewrite DNR rules or data revision state;
- repository, source, parent, temp-root, and filesystem-root output paths are rejected by the pure safety policy;
- valid temporary and repository `build*` outputs remain supported;
- placeholder and malformed Google OAuth IDs are rejected;
- an existing output sentinel survives a failed placeholder Google build, proving validation occurs before cleanup.

Static guards preserve the message-result protocol, atomic blocklist result, blocked-page statistics condition, shared OAuth validation, output-path guard, package test chain, and dedicated Windows workflow inclusion.

## Merge gate

The exact PR head must pass every historical regression through Round 23, strict TypeScript typecheck, ordinary full and blocker-only builds under inherited OAuth environment state, explicit Google build, generated-output validation, cleanup, and post-steps before merge.

## External boundary

Chrome Web Store review, production Google OAuth/API behavior, physical multi-device Browser Sync, browser-specific native-new-tab behavior, and physical accessibility/device testing remain release-environment checks. They are not represented as repository-internal automated proof.
