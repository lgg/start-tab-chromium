# Deep audit round 20 — full-capacity layout and bounded recovery

## Scope

Independent audit of the current `master` after PR #96. The pass mapped every feature and completed-roadmap claim in README and release documentation to production modules, persisted schemas, migrations, runtime ownership, UI actions, build profiles, and executable validation.

Reviewed areas:

- blocklist, DNR ownership, popup, blocked page, countdown and focus statistics;
- all instance-based block types and singleton rules;
- Grid/Free layout editing, presets, zones, pointer/keyboard controls and persistence;
- themes/backgrounds and per-instance settings;
- notes, tasks, links, clocks, alarms, notifications and runtime concurrency;
- local JSON backup, recovery, rollback, Browser Sync, Drive and Calendar;
- schema validation, future-schema protection, migration and collection capacities;
- Manifest V3 permissions, full/blocker-only/Google build profiles and Windows self-hosted CI.

## Material findings and fixes

### 1. Grid capacity was lower than the documented block capacity

Settings normalization limited Grid rows to 500 and both persistence and the inline editor stopped collision resolution after 500 attempts. A dense one-column layout with the supported 1,000 blocks could therefore retain overlaps or change geometry unexpectedly after save.

Fix:

- introduced one shared deterministic interval-based Grid placement implementation;
- removed the retry ceiling;
- defined shared geometry limits that can hold all 1,000 maximum-height blocks;
- used the same placement in settings normalization and the inline editor;
- re-enabling blocks, switching to Grid mode and changing a zone now resolve the complete draft before save;
- drag, resize and keyboard editing share the same height and row bounds as persistence.

### 2. Local Tasks did not enforce its documented limits in the UI

Runtime messages correctly rejected more than 10,000 tasks and titles longer than 500 characters, but the task form still allowed those submissions. The form also cleared a new task before persistence succeeded.

Fix:

- shared the 500-character title limit with the UI;
- disabled add controls at 10,000 tasks;
- repeated the capacity check at submit time to cover stale UI state;
- retained typed input until the runtime mutation succeeds.

### 3. Backup return-navigation metadata was unbounded and could become stale

`lastBlockedUrls` was normalized as an unrestricted map. Corrupt local state or an untrusted backup could retain thousands of return URLs unrelated to the actual blocklist, increasing backup/sync/storage size even after sites were removed.

Fix:

- strict external imports reject more than 5,000 normalized return URLs before Chrome API access;
- local recovery deterministically bounds the map;
- only entries associated with the normalized current blocklist survive export, import or recovery;
- prototype-like host keys remain ordinary data through null-prototype dictionaries.

## Regression coverage

Round 20 fixtures execute:

- deterministic placement of 1,000 maximum-height blocks in one column;
- no overlap beyond the former 500-row boundary;
- tail-to-start fallback placement;
- settings normalization using the same full-capacity algorithm;
- shared geometry and task-title limits;
- strict rejection of oversized external return-URL maps;
- bounded recovery and removal of stale return URLs;
- prototype-safe host preservation;
- complete backup migration correlation.

Static guards prevent the arbitrary 500-attempt algorithm, duplicated editor placement, task form limit drift, premature input clearing, and uncorrelated backup URL normalization from returning.

## Validation gate

The change is not considered complete until the exact PR head passes the dedicated Windows self-hosted job with:

- all historical regression suites through Round 20;
- strict TypeScript typecheck;
- full extension build;
- blocker-only build;
- Google-enabled build;
- final workspace and project-cache cleanup.

## External boundary

Repository automation cannot prove Chrome Web Store approval, live production Google OAuth/API behavior, physical multi-device Sync propagation, every Chromium derivative's internal new-tab handling, or physical pointer/touch/screen-reader behavior. Those remain reproducible manual release checks and are not represented as automated evidence.
