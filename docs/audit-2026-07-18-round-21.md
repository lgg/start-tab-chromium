# Deep audit round 21 — sync identity, Calendar filtering, and UI capacity

## Scope

Independent audit of the current `master` after Round 20. The pass rebuilt the feature-to-code map from README and release documentation instead of treating previous audit reports as proof.

Reviewed areas:

- blocklist storage, DNR redirects, popup, blocked-page countdown, and focus statistics;
- every block type, singleton/repeatable constraints, per-instance runtime, and Layout Editor behavior;
- themes, backgrounds, settings coverage, and all documented collection capacities;
- timers, stopwatches, Pomodoro state, alarms, notifications, and concurrent runtime writes;
- JSON backup/import, rollback, Browser Sync conflict resolution, and Google Drive restore;
- Google Calendar query/limit behavior and weather provider fallback;
- localization, Manifest V3 permissions, full/blocker-only/Google build profiles, and Windows CI.

## Material findings and fixes

### 1. Browser Sync could ignore user data stored under timestamp-like IDs

The v3 canonical checksum recursively removed every object key named `createdAt` or `updatedAt`. Those names are valid user-controlled block IDs, task-collection IDs, domains, and dictionary keys. A note stored under the instance ID `updatedAt`, for example, could change without changing `contentChecksum`, allowing Smart Sync to report that both devices were already identical.

Fix:

- timestamp removal is now path-aware and applies only to generated timestamp fields on known settings, block, theme, runtime, and task entities;
- user-keyed dictionaries preserve `createdAt` and `updatedAt` as ordinary content;
- pristine-state detection uses the same schema-aware rules;
- the previous global-filter checksum remains available only for verification of snapshots uploaded by earlier v3 code;
- restore accepts current, previous-v3, and original legacy checksum forms, so the fix does not strand existing remote backups.

### 2. Google Calendar title filtering happened after the display limit

The Calendar API request asked for only the configured number of upcoming events. The renderer then applied its title filter to that already-truncated list. A matching event just beyond the first N events was therefore invisible even though the block claimed an independent filter and limit.

Fix:

- the query is sent to Google before the display limit is finalized;
- filtered requests use bounded pagination;
- because Google's `q` can also match descriptions and locations, each page is still narrowed to event-title matches;
- pagination stops when the configured display count is filled, the API ends, a page token repeats, or the safety page limit is reached;
- the renderer delegates the complete query-before-limit operation and includes the query in its request-cache identity.

### 3. Custom-theme capacity was not enforced by the Options UI

Storage correctly rejected a 1,001st custom theme, but Create, Import, and Duplicate controls remained enabled at the documented 1,000-theme boundary. Users could enter an editor or select a file only to fail during persistence.

Fix:

- Create Theme, Import Theme, and every Duplicate Theme control disable at the shared `MAX_CUSTOM_THEMES` boundary;
- the UI displays the reached capacity count;
- persistence validation remains the final race-safe guard.

### 4. Smart Sync hid its actual result

Smart Sync briefly set `uploaded`, `restored`, or `unchanged`, then the generic action wrapper replaced it with `sync complete`. This removed the deterministic conflict-resolution result from the user-visible UI.

Fix:

- the shared action wrapper now carries typed operation results;
- a success message may be derived after reload/render from the actual result;
- Smart Sync retains the specific uploaded/restored/unchanged outcome.

## Regression coverage

Round 21 executable fixtures verify:

- generated entity timestamps remain checksum-neutral;
- a user dictionary key named `updatedAt` changes the current Browser Sync checksum;
- the compatibility checksum reproduces the previous behavior for existing remote snapshots;
- Calendar title filtering continues to a second API page before applying a one-event display limit;
- `q`, bounded page size, and continuation tokens are sent correctly.

Static guards prevent:

- a return to global timestamp-key removal;
- removal of previous-v3 checksum compatibility;
- Calendar filtering of an already-truncated renderer result;
- cache collisions between Calendar blocks with different queries;
- enabled theme create/import/duplicate controls at capacity;
- overwriting the specific Smart Sync result with a generic status.

## Validation gate

The change is complete only when the exact PR head passes the dedicated Windows self-hosted job with:

- all historical regression suites through Round 21;
- strict TypeScript typecheck;
- full extension build;
- blocker-only build;
- Google-enabled build;
- final workspace and project-cache cleanup.

## External boundary

Repository automation cannot prove Chrome Web Store approval, production Google OAuth/API credentials, physical multi-device Sync propagation, every Chromium derivative's native-new-tab implementation, or physical pointer/touch/screen-reader behavior. Those remain explicit manual release checks and are not represented as automated evidence.
