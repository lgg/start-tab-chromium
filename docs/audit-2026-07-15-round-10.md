# Deep Audit Round 10 — Atomic Sync Snapshots, Revision Transactions, and Downgrade Safety

Date: 2026-07-15
Base: `master` at `c1d334de767ebfaa8fd4e17ca641d580c688df23` (merged PR #81).

## Scope

This round independently re-audited the complete production tree after Round 9. The review covered every local-data writer, data-revision ordering, Start Page settings/runtime migration, per-instance cleanup, Chrome alarm ownership, backup import/export, Browser Sync conflict resolution, remote snapshot transfer races, future-schema downgrade behavior, service-worker message paths, all three build profiles, generated manifests, strict TypeScript, static architecture checks, executable failure injection, and production dependency security.

## Material findings fixed

### Atomic backup content and Sync revision

- Browser Sync previously exported storage under the shared `data-write` lock, released that lock, and only then read `startTabDataRevision`.
- A queued mutation could therefore make an old backup payload appear to carry a newer revision.
- `exportBackupSnapshot()` now captures normalized backup content and its revision inside one `data-write` transaction, and all Sync uploads use that atomic pair.

### Export time was incorrectly treated as content modification time

- `backupModifiedAt()` included `exportedAt`, which changes for every export even when user data does not.
- A stale device exporting later could appear newer than a genuinely updated remote device and win smart-sync conflict resolution.
- Export time is now excluded. Current data revision is authoritative; actual settings/runtime timestamps are only a legacy fallback when revision metadata does not yet exist.

### Runtime reset alarm race

- Runtime storage was cleared under `data-write`, but clock alarms were cleared after releasing the lock.
- Another context could start a new timer between those steps and have its new alarm deleted by the older reset operation.
- Runtime reset now holds the shared lock through storage removal, alarm clearing, revision commit, and exact rollback.

### Orphaned per-instance runtime after block deletion

- Settings deletion could happen before runtime cleanup.
- Runtime normalization then no longer knew the deleted block ID and silently discarded it from the in-memory view, causing cleanup to conclude that no runtime existed while raw notes/tasks/clocks and alarms remained in storage.
- Instance deletion now inspects raw current and legacy runtime maps, removes orphaned records, clears matching alarms under the same lock, and restores exact storage/revision/alarm state if any step fails.

### Revision commits were not atomic across ordinary writers

- Settings, runtime notes/tasks/clocks, focus statistics, locale, onboarding, legacy blocklist migration, and remembered blocked URLs wrote user data first and revision metadata second.
- If revision persistence failed, content remained changed while Browser Sync still considered it old.
- A shared `commitStorageMutationWithRevision()` transaction now snapshots exact key presence plus revision, executes the mutation, commits revision, and restores both data and revision on any failure.
- Existing custom transactions for backup, DNR blocklist changes, clock reset, and instance deletion retain their additional DNR/alarm rollback while benefiting from the stricter revision guard.

### Unsupported future data-revision schema

- A newer revision schema was previously treated as missing and could be overwritten by an older extension.
- Read/sync operations now fail closed on future revision metadata, and all older ordinary writers preserve it exactly.
- Explicit complete Start Page reset remains the recovery path. When replacing future revision metadata, it uses the previous numeric timestamp as a monotonic floor rather than moving conflict ordering backwards.

### Unsupported future focus-statistics schema

- A future `focusStats` version normalized to empty v1 data and could be overwritten by the next blocking or Pomodoro event.
- Background statistics writers and backup/sync export now reject unsupported future stats without altering them.
- Explicit user-requested statistics reset may intentionally replace the unsupported stats payload when the general revision schema is still compatible.

### Future Browser Sync metadata and backup schemas

- Sync metadata from a newer protocol version was formerly parsed as absent, allowing an older client to upload over it.
- The same risk existed when the metadata protocol remained version 3 but referenced a future `backupVersion`.
- Upload, restore, metadata reads, and smart sync now reject both conditions for local and remote metadata, including a second check immediately before remote commit.

### Remote snapshot changed during transfer

- Another device could replace remote metadata after this device read the chunk list but before local import, allowing a stale snapshot to be restored.
- Restore now re-reads and compares the full metadata identity after chunk/checksum validation and before import.
- Upload similarly re-reads the committed metadata before updating the local baseline; a concurrent replacement fails closed instead of recording a false successful baseline.

## Executable regression coverage

Round 6 and Round 7 fixtures now verify:

- backup content and revision are captured atomically while a settings mutation waits on the same lock;
- repeated exports without user changes retain the same content revision;
- settings, runtime, focus stats, locale, and remembered blocked-URL writes restore exact state when revision persistence fails after applying;
- runtime reset restores exact runtime, legacy data, revision, and alarm metadata on failure;
- orphaned raw instance runtime and alarms are deleted after the settings record is already gone;
- failed orphan cleanup restores exact runtime and alarm state;
- future data-revision metadata blocks old mutations and Sync without being altered;
- complete reset replaces future revision metadata monotonically;
- future focus-statistics schemas survive background writes and export attempts;
- future local/remote Sync protocol metadata survives upload, restore, and smart-sync attempts;
- current Sync protocol metadata carrying a future backup version is also protected;
- remote metadata replacement during chunk transfer aborts before local import;
- prior Round 6/7 migration, alarm, rollback, build, and release fixtures remain green.

## Independent audit passes

### Pass 1

The first pass reviewed runtime ownership, backup/sync ordering, reset behavior, orphan cleanup, future schemas, and the existing executable fixtures. It produced the atomic snapshot, alarm-locking, orphan cleanup, content timestamp, future stats, and future Sync metadata fixes.

### Pass 2

The second pass re-read every changed production path as a transaction graph. It identified the cross-module revision rollback gap, future revision schema overwrite, future backup version carried by the current Sync protocol, and remote metadata time-of-check/time-of-use races. These were fixed and covered with additional dynamic failure/concurrency fixtures and static architecture assertions.

## External validation boundary

Repository automation cannot prove Chrome Web Store review, real production Google OAuth/API behavior, true multi-device propagation and quota behavior, browser-specific native-new-tab restrictions, or physical pointer/touch/high-DPI/screen-reader behavior. Those remain explicit interactive deployment checks and are not represented as automated evidence.
