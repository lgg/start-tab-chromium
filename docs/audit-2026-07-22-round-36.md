# Deep audit round 36 — independent full-project adversarial review

## Scope

Independent review of the exact `master` state produced by PR #112. The pass re-inspected every tracked source/configuration file and all user-facing surfaces: New Tab, onboarding, layout editor, Options, popup, blocked page, early disabled/Split View gate, service worker, settings/runtime storage, backups, Browser Sync, Google Drive, localization, build profiles, documentation, and the historical validation chain.

## Confirmed defects

1. Browser Sync upload confirmed only the committed metadata after `chrome.storage.sync.set`. A cross-device item-level race could leave metadata from the current upload while one or more chunks belonged to another write; the local sync baseline was then advanced without proving the remote frame was readable.
2. Applying a layout preset from Options could immediately overwrite the preset's mode and column count with stale form controls. In particular, applying a grid preset while currently in free layout could keep the page in free mode and make the preset appear to do nothing.
3. Preset profile metadata was derived from the selector rather than the final geometry, so manually changed layouts could remain mislabeled as built-in presets.
4. Options navigation was created only after asynchronous statistics loading. A statistics or migration read failure therefore removed navigation for every otherwise functional settings section.
5. Any Start Tab page with an opener tab was treated as a Split View picker, even without a browser-specific Split View marker. Ordinary extension-opened pages could therefore display the wrong full-screen picker.
6. The early disabled/Split View gate used only the browser UI locale and ignored the extension's explicit English/Russian locale override.
7. The early gate and onboarding were visually modal but did not completely coordinate focus containment, focus restoration, or shared `inert` ownership.
8. Because the gate resolves asynchronously, onboarding could open before the disabled/Split View decision completed, leaving two modal overlays stacked on the same page.
9. Direct JSON import, recovery restore, Browser Sync restore, and Google Drive restore replaced local data immediately without a final destructive-action confirmation.

## Corrections

- Browser Sync upload now verifies the full committed remote bundle, including chunk completeness/checksum/content checksum and a final unchanged-metadata read, before writing local sync metadata.
- Preset application explicitly restores grid mode and preset columns. Options synchronizes visible controls, invalidates the selector after manual geometry changes, and retains a preset profile only when final geometry exactly matches it.
- Options navigation renders synchronously. Statistics failure now produces a localized fallback section while all other settings remain available.
- Split View detection now requires explicit markers in the current context or opener URL/pending URL/title; a generic opener relationship is insufficient.
- The early gate loads the selected locale catalog, traps keyboard focus, restores prior focus, and shares inert state with onboarding.
- A gate readiness promise prevents onboarding from racing or stacking with disabled/Split View overlays.
- Every direct destructive restore/import action now asks for confirmation before replacing local data; smart sync retains its documented automatic latest-wins behavior.
- Added Round 36 executable fixtures, static contracts, full CI wiring, updated historical Round 35 compatibility assertions, and expanded manual QA.

## Validation boundary

Repository automation can prove source contracts, deterministic Browser Sync fixtures, settings transformations, locale parity, type safety, and all package builds. It cannot prove browser-specific Split View marker behavior, physical keyboard focus in every Chromium fork, live Google/third-party services, production OAuth, Chrome Web Store review, or real cross-device conflict timing. Those remain explicit manual release checks.
