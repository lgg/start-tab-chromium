# Post-Round 29 deep audit — DNR namespace, shared quota preflight, and complete Calendar pagination

## Scope

Independent adversarial review of the exact `master` state produced by PR #106. The pass rechecked the new dynamic-rule ownership model under real coexistence constraints, the complete Google Calendar page contract, and whether the self-hosted CI contract still enumerated every mandatory regression suite.

## Material findings and fixes

### 1. Safe DNR collision handling still disabled the blocklist

Round 29 correctly stopped deleting unrelated dynamic rules, but generated blocklist IDs still started at `1`. If another Start Tab feature legitimately used one of those IDs, synchronization failed before mutation. User data remained safe, but the requested site was not blocked.

Fix:

- blocklist rules now use the dedicated range beginning at `1,000,000`;
- unrelated low-ID rules are preserved and no longer prevent synchronization;
- exact legacy redirects from earlier versions are recognized and migrated out of the low range;
- snapshots and rollback understand both the current namespace and the bounded legacy shape;
- foreign-rule collision detection remains as a final atomic safety guard.

### 2. The advertised blocklist capacity ignored shared Chrome quotas

`MAX_BLOCKED_SITES` reflects Chrome's unsafe dynamic-rule ceiling, but that ceiling is shared by every feature in the same extension. A list at the local limit plus one unrelated redirect rule passed internal validation and then failed inside `updateDynamicRules()`.

Fix:

- replacement computes the final foreign-plus-blocklist dynamic rule set before mutation;
- Chrome's exposed total and unsafe dynamic-rule limits are used when available, with conservative compatibility fallbacks;
- unsafe actions are counted separately from Chrome's safe action types;
- over-capacity operations fail with a precise error before the atomic API call;
- existing rules remain untouched after rejection.

### 3. Calendar pagination still stopped after the first page without `q`

Round 29 stabilized `timeMin`, but the loop still treated an empty search query as proof that one page was complete. Google Calendar explicitly permits a page containing fewer than `maxResults`, or even zero items, while returning a non-empty `nextPageToken`.

Fix:

- every non-empty, unseen `nextPageToken` is followed whether or not `q` is present;
- all pages continue to reuse the exact original `timeMin` and request parameters;
- the configured result limit, page cap, and cycle protection remain intact.

### 4. The self-hosted CI contract stopped enumerating new regressions after Round 26

The workflow and package test chain ran Rounds 27–29, but `validate-self-hosted-ci.mjs` no longer listed them in its required regression command set.

Fix:

- the contract now explicitly requires executable and static validation for Rounds 27, 28, 29, and 30;
- file-existence and workflow-command checks cover the complete current regression chain.

## Regression coverage

Round 30 executable fixtures verify:

- a foreign rule with ID `1` survives while the requested blocklist rule is installed;

Static contracts preserve the namespace, migration ownership predicate, quota preflight order, unconditional page-token handling, updated historical fixture expectation, complete package/CI wiring, and the full self-hosted regression command list.
