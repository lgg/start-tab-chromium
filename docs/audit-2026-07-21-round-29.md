# Post-Round 28 deep audit — isolated DNR ownership, complete instance rollback, and stable Calendar pagination

## Scope

Independent adversarial review of the exact `master` state produced by PR #105. This pass focused on shared Chrome API surfaces that can be used by multiple extension features, rollback paths not exercised by Round 28, and multi-page Google Calendar request consistency.

## Material findings and fixes

### 1. Blocklist synchronization treated every dynamic rule as its own

The blocklist snapshot and replacement helpers read the complete `chrome.declarativeNetRequest` dynamic rule set. A normal blocklist sync therefore removed unrelated dynamic rules created by another Start Tab feature. A foreign rule using one of the blocklist's numeric IDs could also be silently replaced.

Fix:

- blocklist ownership is identified by the extension redirect target for `blocked.html?site=`;
- snapshots and rollback contain only blocklist-owned rules;
- synchronization removes only currently owned blocklist rules;
- unrelated dynamic rules remain untouched;
- a foreign rule-ID collision is detected before `updateDynamicRules()` and fails without mutating the rule set.

### 2. Instance-runtime deletion still had sequential rollback

`deleteInstanceRuntime()` restored storage before alarms. If storage restoration failed, durable alarm restoration was never attempted, even though reset and general runtime transactions had already moved to independent rollback effects.

Fix:

- instance storage and alarm restoration now execute as independent best-effort effects;
- alarm recovery is attempted even when storage recovery fails;
- the original operation error and incomplete rollback remain visible through an aggregate error.

### 3. Calendar pagination changed request parameters between pages

`listCalendarEvents()` generated `timeMin` inside its pagination loop. A later request carrying `nextPageToken` could therefore use a different lower bound from the first page. Google pagination tokens are tied to the original query, so changing parameters can reject a valid token or produce an inconsistent page sequence.

Fix:

- one `timeMin` value is captured before the pagination loop;
- every page of the same logical query reuses that exact lower bound;
- title-only client filtering and bounded pagination behavior remain unchanged.

## Regression coverage

Round 29 executable fixtures verify:

- unrelated dynamic rules surviving blocklist synchronization;
- foreign rule-ID collisions failing before any DNR mutation;
- the complete foreign rule set remaining unchanged after a collision;
- instance alarm rollback still running after storage rollback failure;
- a paginated Calendar query reaching a later title match;
- every page using the exact same `timeMin` together with the next-page token.

Static contracts preserve explicit DNR ownership, foreign collision detection, owned-only removals, independent instance rollback, stable Calendar pagination parameters, package test-chain inclusion, and dedicated Windows CI steps.
