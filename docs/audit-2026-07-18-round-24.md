# Post-Round 23 deep audit — link-safe cleanup and canonical no-op writes

## Scope

Independent review of the production and validation changes merged by PR #100. The audit followed the new output-path guard through real filesystem resolution on Windows and POSIX, reviewed blocklist mutation semantics against the shared data revision and DNR transaction model, and checked all published Google OAuth examples against the validator they instruct users to run.

## Material findings and fixes

### 1. Lexical output containment did not stop symlink or Windows-junction traversal

Round 23 rejected dangerous path strings, but recursive cleanup still accepted a path such as `build-link/nested` when `build-link` was an existing symbolic link or Windows junction to a directory outside the repository. The string remained repository-local while filesystem traversal reached external data.

Fix:

- the builder keeps the existing lexical repository/temp-root policy;
- before recursive removal, every existing path component below the trusted root is inspected with `lstat`;
- symbolic links and Windows junctions are rejected before cleanup;
- missing descendants remain valid because a new normal output tree may be created there;
- the executable regression creates a real junction on Windows and a directory symlink on POSIX, then proves an external sentinel survives the rejected build.

### 2. Canonical no-op blocklist operations advanced the shared sync revision

Adding an already blocked host, replacing the list with the identical canonical list, clearing an already empty list, or recording the same blocked navigation URL again still rewrote storage and/or DNR state and called the data-revision writer. This could trigger unnecessary Browser Sync conflict decisions and repeated DNR churn even though user-visible data had not changed.

Fix:

- transactional blocklist mutations compare exact canonical stored sites and last-URL side data before writing;
- a true canonical no-op returns before storage, DNR, or revision changes;
- malformed or noncanonical stored data is still repaired because the comparison uses exact raw representation rather than normalized equivalence alone;
- repeated identical blocked-navigation metadata returns before its revisioned storage write.

### 3. Published OAuth example was accepted as a valid client ID

README, deployment, and release documentation showed `1234567890-example.apps.googleusercontent.com`. The value looked illustrative but passed the same format and placeholder validation that the documentation claimed would reject examples. Copying it could produce a package that built successfully but could never authenticate.

Fix:

- common placeholder tokens such as `EXAMPLE`, `YOUR`, `CHANGEME`, and `PLACEHOLDER` are rejected in addition to `REPLACE` and `TODO`;
- user-facing commands now show `REPLACE_WITH_REAL_CLIENT_ID.apps.googleusercontent.com`;
- every relevant document explicitly says that the shown value is intentionally rejected and must be replaced entirely;
- executable coverage proves the historical copyable fake ID fails before an existing output directory is cleaned.

## Regression coverage

Round 24 executable fixtures verify:

- duplicate block and identical-list replacement perform no storage, DNR, or revision writes;
- clearing an already empty canonical blocklist remains a true no-op;
- recording the identical blocked URL twice does not advance the revision;
- common OAuth placeholders and the former documentation example are rejected while the dedicated CI validation ID remains accepted;
- failed placeholder validation preserves an existing output sentinel;
- real Windows-junction/POSIX-symlink traversal is rejected before recursive cleanup and external data survives.

Static guards preserve the filesystem-aware cleanup ordering, canonical no-op transaction boundary, OAuth documentation contract, package test chain, and dedicated Windows workflow inclusion.

## Merge gate

The exact PR head must pass every historical regression through Round 24, strict TypeScript typecheck, ordinary full and blocker-only builds under inherited OAuth environment state, explicit Google build, generated-output validation, cleanup, and post-steps on the dedicated Windows runner before merge.

## External boundary

Chrome Web Store review, production Google OAuth/API behavior, physical multi-device Browser Sync, browser-specific native-new-tab behavior, and physical accessibility/device testing remain release-environment checks. They are not represented as repository-internal automated proof.
