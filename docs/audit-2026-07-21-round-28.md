# Post-Round 27 deep audit — strict nested-domain priority and complete runtime rollback

## Scope

Independent adversarial review of the exact `master` state produced by PR #104. This pass focused on DNR specificity at unusual domain depths and on rollback behavior across runtime storage and durable Chrome alarms.

## Material findings and fixes

### 1. One-label blocked parents could still tie with their children

Round 27 used `max(1, labelDepth - 1)` for redirect-rule priority. That worked for ordinary multi-label web domains, but it assigned the same priority to a one-label parent and its first child, for example `localhost` and `app.localhost`. JavaScript selected the most-specific suffix while Chrome DNR could still choose either equal-priority redirect.

Fix:

- generated DNR priority now equals normalized domain label depth;
- every proper child suffix therefore receives a strictly higher priority than its parent;
- the JavaScript matcher and Chrome rule selection remain aligned for ordinary domains, internal domains, and one-label parents;
- historical Round 27 fixtures and documentation were corrected to represent the stronger invariant.

### 2. Runtime storage failure could skip alarm rollback

Clock/runtime transactions, runtime reset, and complete Start Tab reset restored storage before alarms. A storage rollback failure prevented the durable-alarm restoration attempt.

Fix:

- storage and alarm restoration execute through independent best-effort effects;
- both surfaces are attempted even when either one fails;
- incomplete rollback remains visible through the original operation error plus rollback failure details.

### 3. Runtime snapshot writes could be skipped after removal failure

The runtime storage helper removed keys absent from the snapshot and only then restored present values. A failed remove prevented the snapshot set, leaving values written by the failed operation in storage.

Fix:

- absent-key removal and snapshot writes are independent effects;
- exact values are still restored when removal of another key fails;
- all rollback failures remain reported.

### 4. Alarm snapshot restoration stopped on the first failed alarm

Alarm restoration cleared all Start Tab alarms and recreated the snapshot sequentially. One failed clear skipped later clears and all recreation; one failed create skipped later snapshot alarms.

Fix:

- every existing Start Tab alarm receives a clear attempt;
- every snapshot alarm receives a recreation attempt;
- unrelated extension alarms are never touched;
- cleanup and recreation failures are collected and surfaced after all possible recovery work is attempted.

## Regression coverage

Round 28 executable fixtures verify:

- strict DNR priority for `localhost` and `app.localhost`;
- agreement between DNR ownership and JavaScript most-specific matching;
- runtime snapshot restoration after an absent-key removal failure;
- alarm rollback still running after storage rollback failure;
- every relevant clear and every snapshot recreation being attempted despite sibling failures;
- unrelated alarms remaining untouched;
- later snapshot alarms being recreated after an earlier create failure.

Static contracts preserve the full-label priority formula, independent runtime rollback, independent reset rollback, complete alarm restoration, package test-chain inclusion, and the dedicated Windows workflow steps.
