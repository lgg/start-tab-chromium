# Deep audit round 34 — post-Round 33 regression hardening

## Scope

Independent adversarial review of the exact `master` state produced by PR #110. The pass concentrated on the behavior changed in Round 33 and its neighboring state boundaries: note autosave, runtime storage notifications, browser-derived URL filtering, weather geocoding, locale/settings persistence, search-provider editing, backup-compatible runtime normalization, and the complete historical CI chain.

## Confirmed defects

1. Optimistic note, task, and link-page writes were echoed back through `chrome.storage.onChanged` and treated as external runtime changes. The resulting full state refresh could rebuild the Start Tab after every autosave, disrupting textarea focus and cursor position.
2. A naive runtime equality check could still rebuild the page when two semantically identical runtime dictionaries had different key insertion order.
3. Recent History asked Chrome for exactly the visible result count and only then removed privileged/non-web URLs. Unsafe leading entries therefore consumed the limit and could leave the block partly empty even when valid history entries followed.
4. Renaming the currently selected search provider changed its ID and made the default-provider selector fall back to the first row rather than keeping the edited provider selected.
5. Locale persistence occurred before the settings write. A settings conflict or write failure could leave the new locale stored but not applied. Locale-only saves also rewrote settings/runtime revisions unnecessarily, and a locale failure after a successful settings write left the Options page with stale revisions.
6. Custom geocoding endpoints were trusted to return in-range finite coordinates. Invalid latitude/longitude values could be forwarded to the weather provider.
7. Browser-derived title and result-limit edge cases were not normalized: whitespace-only titles did not consistently fall back to the URL, and zero/non-finite limits could produce unexpected output.

## Corrections

- Suppress full runtime refreshes for semantically identical storage echoes while still adopting the newer runtime revision; future runtime schemas remain protected from normalization.
- Compare normalized runtime dictionaries in stable key order.
- Use bounded history overscan, filter through the shared HTTP(S) boundary, and apply the visible limit afterward.
- Preserve provider selection by stable ID, edited row position, or the nearest surviving row.
- Skip no-op settings writes, commit settings before locale changes, reload only after successful locale persistence, and refresh in-memory revisions after a partial settings-only success.
- Validate custom geocoder latitude/longitude type, finiteness, and legal ranges.
- Normalize browser titles and invalid result limits.
- Add executable Round 34 fixtures, static contracts, and full CI wiring.

## External boundary

Repository automation cannot prove physical focus behavior in every Chromium build, real browser history ordering, live third-party weather/geocoding availability, production Google OAuth/API behavior, Chrome Web Store review, or cross-device interaction quality. These remain manual release checks and are not represented as automated proof.
