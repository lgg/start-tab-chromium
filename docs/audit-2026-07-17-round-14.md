## Material findings fixed

### Alarm creation Promises were not awaited

Chrome's Manifest V3 `chrome.alarms.create()` API returns a Promise and can reject. Three durable-alarm paths called it without awaiting the result. As a consequence, code could report a successful import, reconciliation, or clock action before Chrome had accepted the alarm, and an asynchronous rejection could become detached from the transaction.

Every low-level alarm creation is now awaited. The alarm mocks also return real Promises and inject asynchronous rejection instead of only synchronous exceptions, so the regression suite models the browser contract accurately.

### Clock runtime and durable alarms were not one recoverable unit

Starting, pausing, resetting, or completing a clock persisted runtime first and reconciled its alarm afterward. Even after making alarm creation awaitable, a rejected alarm could leave a running Timer or auto-started Pomodoro phase stored without a durable completion alarm.

Clock mutations now commit runtime, data revision, and the complete desired clock-alarm set under the shared `data-write` lock. The exact previous runtime/revision storage and alarm set are restored if reconciliation fails. Alarm reconciliation itself also restores the previous alarm set after partial clear/create failure.

The same transaction is used by direct clock actions and completion/auto-start transitions. Failure-injection verifies exact storage, revision, and alarm rollback for both paths.

### A focus-stat failure suppressed an independent completion notification

Clock completion updated focus statistics before creating the optional notification. If the statistics write failed, notification creation was skipped even though the clock had already completed successfully.

Statistics and notification effects are now attempted independently. Each failure remains observable, multiple failures are aggregated, and a statistics error no longer prevents the user from receiving the configured completion notification.

### Runtime conflict recovery could hide both failures

Optimistic note, task, and link-page writes reload canonical runtime after a conflict. If that recovery read also failed, the conflict announcement and render request were skipped, while several block-level callers intentionally discarded the already-handled mutation rejection.

A dedicated recovery helper now always announces the conflict. Successful recovery queues a canonical render; failed recovery queues a refresh retry and throws an `AggregateError` retaining both the mutation and recovery failures.

## Regression coverage

Round 14 coverage verifies:

- all low-level `chrome.alarms.create()` calls are awaited;
- alarm mocks use asynchronous Promise rejection;
- a rejected clock-action alarm restores exact runtime, revision, and alarms;
- a rejected auto-start completion alarm restores exact runtime, revision, and alarms;
- a delayed schedule re-reads canonical runtime rather than creating a stale alarm;
- a statistics write failure does not suppress the independent completion notification;
- successful runtime-conflict recovery refreshes, announces, and renders in order;
- failed recovery still announces, queues a refresh retry, and preserves both errors;
- static guards prevent non-atomic clock mutation ownership from returning.

## External boundary

Repository automation can validate source, deterministic mocks, asynchronous failure injection, storage and alarm rollback, schemas, migrations, generated packages, TypeScript, localization parity, and dependency advisories. It cannot prove Chrome Web Store approval, production Google OAuth consent and live account behavior, actual multi-device Sync propagation/throttling, every Chromium-derived browser's native-new-tab behavior, or physical pointer/touch/high-DPI/screen-reader interaction. Those remain explicit interactive release checks rather than falsely claimed automated proof.
