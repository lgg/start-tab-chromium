# Post-Round 24 deep audit — bounded CI cleanup and derived-state repair

## Scope

Independent review of the production, regression, and self-hosted Windows CI changes merged by PR #101. This pass followed link safety beyond the extension builder into every repository cleanup path, checked canonical blocklist no-op behavior against the actual Chrome DNR state, and reviewed raw remembered-navigation metadata rather than only its normalized in-memory representation.

## Material findings and fixes

### 1. Self-hosted CI cleanup still used lexical-only recursive deletion

PR #101 protected `build.mjs`, but the workflow still removed workspace and cache paths with PowerShell `Remove-Item -Recurse` after checking only a string prefix. A final Windows junction or an intermediate reparse point could therefore make cleanup semantics differ from the validated lexical path and put external runner data at risk.

Fix:

- one shared Node path-safety module now owns strict descendant validation, `lstat` inspection, and bounded removal;
- intermediate symbolic links and Windows junctions are rejected;
- a final link is removed as the link itself without traversing its target;
- `build.mjs`, local `npm run clean`, and self-hosted CI cleanup reuse the same implementation;
- the project cache must exactly match `RUNNER_TEMP/start-tab-chromium-cache`;
- the workflow no longer creates a possibly stale cache path before inspecting it and no longer contains recursive PowerShell deletion;
- the checked-in cleanup runs after checkout and the pinned Node.js setup, before cache restoration, and again under `if: always()`.

### 2. A canonical storage no-op could leave Chrome DNR state broken

Round 24 correctly avoided storage and revision writes when the requested blocklist already matched canonical storage. It returned before checking the derived Chrome dynamic rules. If a rule had disappeared or been corrupted, repeating “block this site” reported success while navigation remained unblocked until a later startup or explicit synchronization.

Fix:

- canonical storage no-ops now compare the complete expected DNR rule set with the current rules;
- an exact storage-and-DNR match remains a true zero-write no-op;
- missing, extra, or modified rules are rebuilt immediately;
- DNR-only repair does not rewrite user storage and does not advance the shared Browser Sync data revision because the durable user data did not change;
- update failures still propagate instead of being reported as success.

### 3. Normalized remembered-URL comparisons could preserve malformed raw storage

The duplicate-URL optimization read `lastBlockedUrls` through its normalizer and returned early when the normalized value matched. Raw aliases such as `WWW.EXAMPLE.COM`, invalid entries, or an empty object could therefore remain indefinitely. Clearing the final URL also wrote `{}` instead of restoring the canonical absent-key representation.

Fix:

- remembered-navigation operations retain both the raw storage snapshot and normalized dictionary;
- identical values skip writing only when the raw representation is already exact and canonical;
- aliases and invalid entries are rewritten to one canonical host key;
- clearing the final remembered URL removes `lastBlockedUrls` entirely;
- stale empty or malformed metadata is repaired through the revisioned rollback-safe storage helper.

## Regression coverage

Round 25 executable fixtures verify:

- a canonical blocklist plus canonical DNR remains a true no-op;
- a missing DNR rule is restored by a repeated block operation without storage writes or a data-revision bump;
- noncanonical raw remembered-URL data is repaired even when its normalized URL is unchanged;
- the subsequent exact canonical duplicate remains a no-op;
- final and stale-empty remembered-URL states remove the storage key;
- the real CI cleanup removes ordinary targets and a final Windows junction/POSIX symlink while preserving the external sentinel and unrelated workspace files.

Static guards preserve the shared path-safety implementation, cleanup ordering, exact cache boundary, absence of recursive PowerShell deletion, DNR self-healing contract, raw metadata canonicalization, package test chain, and dedicated Windows workflow inclusion.

## Merge gate

The exact PR head must pass every historical regression through Round 25, strict TypeScript typecheck, ordinary full and blocker-only builds under inherited OAuth environment state, the explicit Google build, generated-output validation, junction-safe cleanup, and all action post-steps on the dedicated Windows runner before merge.

## External boundary

Chrome Web Store review, production Google OAuth/API behavior, physical multi-device Browser Sync, browser-specific native-new-tab behavior, physical accessibility/device testing, and operating-system-level attacks that replace filesystem components concurrently between validation and deletion remain outside repository-internal automated proof.
