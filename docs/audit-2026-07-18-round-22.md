# Deep audit round 22 — deterministic build profiles and guarded blocked-page state

## Scope

Independent continuation from the Round 21 `master`. This pass rechecked release-profile boundaries and user flows that cross generated artifacts, extension pages, the service worker, and persisted blocker state.

Reviewed areas included:

- full, blocker-only, and Google-enabled build selection;
- generated manifests, permissions, omitted assets, and environment isolation;
- Options behavior in each build profile;
- blocked-page countdown, stale tabs, direct extension-page navigation, and unblock statistics;
- release/deployment documentation and the Windows self-hosted CI contract;
- all earlier regression suites and production builds.

## Material findings and fixes

### 1. Ordinary builds could inherit Google OAuth accidentally

The builder treated any non-empty `GOOGLE_OAUTH_CLIENT_ID` environment value as permission to inject OAuth configuration and `identity`. This meant `npm run build` and even `npm run build:blocker-only` could silently change profile when launched from a shell, runner, or host service that already contained that variable.

That contradicted the documented product profiles: ordinary full and blocker-only builds are Google-disabled, while `npm run build:google` is the explicit Google-enabled profile.

Fix:

- the builder now requires an explicit `--google` profile flag before it reads or injects the OAuth client ID;
- ordinary full and blocker-only builds always strip `oauth2` and `identity`, even when an OAuth environment value is inherited;
- Google mode requires a valid client ID and cannot be combined with blocker-only mode;
- generated-output validation now distinguishes `full`, `blocker-only`, and `google` rather than inferring the profile from ambient environment state;
- CI deliberately supplies a fake inherited client ID to ordinary builds and proves that it is ignored.

### 2. Blocker-only Options exposed unavailable Start Tab controls

The blocker-only artifact intentionally omits `newtab.html` and the manifest override, but Options still rendered an Open Start Tab button targeting that absent file. It also showed an enabled Start Tab content checkbox even though the selected artifact could not own the browser new-tab page.

Fix:

- Options derives Start Tab availability from the generated manifest;
- the Open Start Tab action is omitted when `chrome_url_overrides.newtab` is absent;
- the Start Tab content checkbox is disabled in blocker-only builds;
- saving unrelated settings preserves the latent Start Tab preference so switching back to a full build does not unexpectedly rewrite it.

### 3. Stale or directly opened blocked pages could perform a fake unblock flow

A valid extension-owned `blocked.html?site=...` URL could remain open after another context had already removed the site, or could be opened directly. The page accepted the syntactically valid host, displayed the countdown, sent an idempotent unblock request, and then recorded a countdown unblock even though no active block had been removed.

Fix:

- startup confirms that the exact host is still represented by the current blocklist;
- stale/direct pages immediately leave the interstitial instead of offering an unblock action;
- countdown completion rechecks block state to cover changes made while the countdown is running;
- countdown-unblock statistics are recorded only when the page observed an active block and completed the removal path;
- service-worker failures remain distinguishable from an already-resolved stale page.

## Regression coverage

Round 22 executable fixtures build temporary artifacts and verify:

- default full ignores an inherited OAuth client ID;
- blocker-only ignores an inherited OAuth client ID and still omits new-tab/history capabilities;
- explicit Google mode injects the requested client ID and `identity`;
- Google plus blocker-only is rejected;
- explicit Google mode without a client ID is rejected.

Static guards verify:

- package scripts retain explicit Google profile selection;
- build and output validators cannot regress to ambient-environment profile inference;
- blocker-only Options cannot expose the missing new-tab action or rewrite latent Start Tab state;
- blocked-page startup and countdown both recheck active block state;
- unblock statistics remain conditional on the actual removal path;
- deployment and release documentation match production behavior;
- Round 22 remains in both `npm test` and the dedicated Windows workflow.

## Merge gate

The change is complete only after the exact PR head passes the dedicated `start-tab-chromium-ci` job with:

- every historical regression suite through Round 22;
- strict TypeScript typecheck;
- ordinary full build under an inherited OAuth variable;
- blocker-only build under an inherited OAuth variable;
- explicit Google-enabled build;
- generated-output validation for all profiles;
- final workspace/cache cleanup and post-steps.

## External boundary

Repository automation still cannot prove Chrome Web Store approval, production Google OAuth consent and API behavior, physical multi-device Browser Sync, every Chromium derivative's native-new-tab behavior, or physical accessibility/device interaction. Those remain explicit release-environment checks rather than incomplete repository work.
