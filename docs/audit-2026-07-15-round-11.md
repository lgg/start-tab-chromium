Date: 2026-07-15

## Scope

This independent pass re-audited the completed 3.0.0 roadmap against the current production tree instead of relying on roadmap checkboxes or previous PR descriptions. The review covered versioned backup migration, legacy storage compatibility, backup rollback inventory, Google Calendar event semantics, Google Drive transport framing, generated build profiles, strict TypeScript, static architecture checks, and executable migration/rendering fixtures.

## Material findings fixed

### Legacy blocklists were lost when importing old backups

The live extension migrates the historical `blocked` storage key, but backup versions 1‚Äì4 were normalized only from `blockedSites`. An old valid backup containing `blocked` could therefore import successfully while silently losing its blocklist.

The backup schema now:

- captures the historical `blocked` key in source and rollback snapshots;
- merges `blockedSites` and `blocked` during export/migration;
- applies the same host normalization, `www.` removal, deduplication, and sorting used by the live blocklist;
- emits only the current `blockedSites` key;
- removes the obsolete key after a successful import while restoring it exactly on rollback.

### Google Calendar all-day events drifted across time zones

Google date-only events were flattened into ordinary strings and later parsed as timestamps. Depending on the local time zone, an all-day event could display on the previous date and include an artificial clock time.

Calendar events now preserve an explicit `allDay` marker. The renderer parses date-only values as local calendar dates, renders a localized date plus the existing localized ‚ÄúAll day‚Äù label, and never renders a clock time for those events. Timed events retain localized date and time output.

### Google Drive multipart uploads used a fixed boundary

New Drive backup files used the constant multipart boundary `start-tab-boundary`. A valid JSON backup containing that exact byte sequence could collide with the MIME framing.

Multipart uploads now generate a UUID-based boundary and verify that it does not occur in either metadata or backup JSON before constructing the request.

### Successful empty Google API responses were treated as invalid JSON

The Google API helper always called `response.json()`. Successful media update responses with an empty body could therefore fail after the remote backup had already been written.

The helper now reads the response body as text and returns `undefined` for a successful empty body, while still parsing normal JSON responses and preserving existing error handling and token retry behavior.

## Regression coverage

Round 11 adds executable and static validation for:

- merging and deduplicating `blockedSites` plus legacy `blocked` backup data;
- dropping the obsolete key from migrated output;
- all-day Calendar labels containing a localized date and no clock time;
- timed Calendar labels retaining a clock time;
- retention of an explicit all-day marker from Google payloads;
- UUID-based Drive multipart boundaries and rejection of the old fixed boundary;
- inclusion of this validation in the default `npm test` suite.

## Validation boundary

The pull request runs the complete existing validation graph: static and migration fixtures, strict TypeScript, full build, blocker-only build, and Google-enabled build. Real production OAuth consent, Google Calendar/Drive network behavior, Chrome Web Store review, multi-device sync propagation, and physical browser interaction remain external checks and are not represented as automated proof.
