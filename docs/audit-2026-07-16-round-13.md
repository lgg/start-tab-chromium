# Deep Audit Round 13 — Mixed Storage Events, Sync Total Quotas, Drive Determinism, and Visible UI Failures

Date: 2026-07-16
Base: current `master` after merged PR #84.

## Scope

## Material findings fixed

### Mixed storage events could leave runtime state stale while editing a layout

The new-tab storage listener intentionally ignored external settings writes while the Layout Editor had an unsaved draft. However, it returned from the complete event. If the same storage transaction changed settings together with runtime or focus statistics, the independent runtime/statistics refresh was also discarded.

Storage changes now use a pure planner. An unsaved settings draft remains protected and produces the existing warning, while runtime and focus changes contained in the same event still refresh before the next render.

### Browser Sync enforced only the per-item quota

The previous audit correctly measured each item using its key and JSON-stringified value. The upload path still did not verify Chrome's total `storage.sync` quota. Existing extension-owned keys or stale chunks could therefore make every individual item valid while the complete write was rejected.

The upload path now:

- reads the runtime `QUOTA_BYTES` limit with the documented fallback;
- calculates the exact eventual sync-area size from every key and JSON-stringified value;
- rejects an oversized snapshot before changing remote data;
- detects temporary quota pressure caused by stale chunks;
- removes those stale chunks only when necessary;
- restores them if the subsequent write fails, reporting an `AggregateError` if rollback is incomplete.

### Duplicate Google Drive backups were selected nondeterministically

The appDataFolder lookup returned the first file from an unordered Drive list. Duplicate `start-tab-backup.json` files can exist after concurrent first uploads or legacy behavior, so restore or update could target an arbitrary older file.

The list request now orders by `modifiedTime desc`, requests `pageSize=1`, and includes `modifiedTime` in the selected fields. Both upload and restore therefore operate on the newest matching backup deterministically.

### Several Start Tab actions hid rejected operations

Clock controls, background clock completion, and Command-block actions used a button helper that discarded rejected promises. A storage, service-worker, alarm, or export failure could leave the UI unchanged without explaining why.

Block render contexts now expose a visible error channel. Direct clock and command actions route failures to the Start Tab status region, while state mutations retain their existing conflict-refresh behavior.

## Regression coverage

Round 13 executable fixtures verify:

- mixed settings/runtime events preserve an unsaved layout draft and still refresh runtime;
- settings-only external writes remain ignored while editing;
- focus-statistics changes refresh independently;
- Drive backup lookup requests the newest single appDataFolder result;
- total Browser Sync quota accounting includes every key and exact JSON stringification.

Static guards prevent removal of total-quota enforcement, recoverable stale-chunk cleanup, deterministic Drive ordering, the mixed-storage planner, and visible block-action error handling.

## External boundary

Repository automation can validate source, generated packages, deterministic API requests, storage mocks, migrations, concurrency/failure fixtures, TypeScript, and dependency advisories. It cannot prove Chrome Web Store approval, real OAuth consent, live Google account behavior, actual multi-device Sync propagation/throttling, every Chromium-derived browser's native-new-tab behavior, or physical pointer/touch/high-DPI/screen-reader interaction. Those remain explicit interactive checks rather than falsely claimed automated results.
