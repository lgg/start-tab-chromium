# Post-Round 26 deep audit — deterministic domain ownership and exact rollback

## Scope

Independent adversarial review of the exact `master` state produced by PR #103. This pass concentrated on overlapping parent/subdomain block entries, navigation-accounting consistency, exact Chrome DNR restoration, and rollback behavior when more than one storage/API surface fails.

## Material findings and fixes

### 1. Parent and nested domains could compete for one navigation

Chrome `requestDomains` rules match the listed domain and its subdomains. The blocklist generated every redirect rule with the same action and priority. If both `example.com` and `app.example.com` were blocked, both rules matched `app.example.com`; equal-priority redirect ordering is not a stable identity contract.

Fix:

- JavaScript matching now explicitly chooses the longest matching blocked suffix;
- generated DNR priority equals normalized domain label depth;
- a more-specific blocked subdomain therefore always wins before its parent rule, including one-label parents such as `localhost`;
- the approach does not create growing exclusion arrays as more nested sites are blocked;
- the blocked-page `site` parameter, unblock operation, remembered URL, and per-domain statistics resolve to the same most-specific site.

### 2. One blocked navigation was matched twice

The service worker first called `blockedSiteForUrl()` for statistics and then called `rememberBlockedNavigation()`, which read the blocklist again. A concurrent mutation between those calls could associate the return URL and statistics with different sites.

Fix:

- `rememberBlockedNavigation()` performs one match under the shared `data-write` lock;
- it persists metadata and returns the selected site from that same snapshot;
- the worker records statistics with the returned site and performs no preliminary blocklist read.

### 3. Failed blocklist changes did not restore exact DNR state

Rollback rebuilt rules from the previous `blockedSites` array. That restored canonical derived state but not the exact rule snapshot that existed before the failed operation. A pre-existing extra or modified rule could disappear even though the user-visible mutation failed.

Fix:

- every blocklist mutation captures exact dynamic rules before changing storage or DNR;
- failed mutations restore that exact snapshot rather than regenerating it;
- DNR-only repair also restores its original snapshot if repair rejects after partial application;
- storage and DNR rollback are independent best-effort effects, so one failure cannot skip the other;
- incomplete restoration is surfaced through `AggregateError`.

### 4. Backup and revision rollback could stop after the first error

Backup rollback restored storage, regenerated DNR, and restored alarms sequentially. A storage restore failure skipped DNR and alarm recovery. Generic revisioned storage rollback similarly skipped snapshot writes when absent-key removal failed.

Fix:

- backup import captures exact pre-import DNR in addition to storage and alarms;
- storage, exact DNR, and alarms are restored as independent effects;
- backup storage set/remove operations independently attempt both halves;
- revisioned absent-key removal and snapshot writes independently attempt both halves;
- all failures remain visible and are never converted into success.

## Regression coverage

Round 27 executable fixtures verify:

- most-specific parent/subdomain matching;
- strictly higher DNR priority for the nested blocked domain;
- one locked site identity for remembered URL persistence;
- exact restoration of an unrelated pre-existing DNR rule after a failed mutation;
- continued DNR restoration after storage rollback failure;
- continued snapshot restoration after absent-key removal failure;
- exact backup restoration of storage, DNR, and durable alarms.

Static contracts preserve the matcher, specificity priority, atomic worker flow, exact DNR snapshots, independent rollback wiring, package test chain, and dedicated Windows workflow steps.
