# Post-Round 25 deep audit — repeatable repair and bounded checkout cleanup

## Scope

Independent adversarial review of the production and CI state merged by PR #102. This pass rechecked the distinction between durable blocklist data and derived Chrome DNR rules, legacy migration lifetime, live remembered-navigation metadata, failure rollback, and every deletion path that runs before the repository's checked-in path-safety module.

## Material findings and fixes

### 1. A no-op unblock could leave an extra Chrome DNR rule active

`unblockHost()` returned `false` immediately when the normalized host was absent from `blockedSites`. That was correct for durable user data but skipped the Round 25 DNR comparison. If Chrome retained an extra redirect rule, the user remained blocked even after repeating the unblock action.

Fix:

- every worker-side unblock now enters the same atomic blocklist mutation path;
- the returned `changed` value still reports only whether durable blocklist data changed;
- an extra, missing, or modified DNR rule is reconciled independently;
- DNR-only repair does not rewrite storage or advance Browser Sync revision;
- a failed DNR-only repair propagates and is never reported as a successful unblock.

### 2. Successful legacy migration was cached for the entire worker lifetime

The migration promise was reset after rejection but retained forever after success. A later `blocked` key introduced by old restore state or another storage path was therefore ignored. Empty or malformed legacy values also returned early and remained in storage indefinitely.

Fix:

- concurrent callers still share one in-flight migration;
- the promise is released in `finally` after both success and failure;
- migration checks key presence rather than non-empty array length;
- empty and malformed legacy containers are removed as stale schema state;
- current and legacy sites are merged canonically;
- transient removal failure restores the exact blocklist and revision snapshot, and the next call can retry.

### 3. Live remembered URLs could outlive their blocked sites

Backup migration already correlated `lastBlockedUrls` with the normalized blocklist, but ordinary runtime mutation only normalized each entry independently. Valid metadata for a no-longer-blocked host could therefore survive and be written back during an unrelated URL update.

Fix:

- one null-prototype filter retains remembered URLs only for active normalized sites;
- transactional blocklist mutations, duplicate remembered-navigation checks, and explicit clears use the same correlation rule;
- raw aliases and malformed entries still receive the Round 25 canonical repair;
- an empty result removes the storage key through the existing revisioned rollback-safe helper.

### 4. Checkout performed a broad clean before bounded cleanup existed

The workflow configured `actions/checkout` with `clean: true`. Its built-in workspace clean ran before checkout completed and before the pinned Node.js toolchain made `path-safety.mjs` available. The checked-in junction-aware remover was therefore not the sole owner of deletion, contrary to the documented narrow cleanup contract.

Fix:

- checkout uses `clean: false`;
- the tracked source tree is still updated by checkout;
- conflicting untracked state must fail visibly instead of being silently deleted;
- after the pinned Node.js setup, `clean-ci.mjs` removes only the documented project outputs and exact project cache;
- the same bounded cleanup repeats under `if: always()`;
- static release and self-hosted CI contracts reject a future return to `clean: true`.

## Regression coverage

Round 26 executable fixtures verify:

- stale extra DNR removal through a no-op unblock;
- DNR repair failure propagation with zero storage or revision writes;
- exact rollback and retry after legacy-key removal failure;
- migration of a legacy key introduced after an earlier successful pass;
- removal of an empty legacy key;
- correlation and cleanup of remembered URLs for no-longer-blocked sites.

Static guards preserve repeatable migration, active-site URL correlation, separate durable-change reporting, checkout `clean: false`, bounded cleanup ordering, package wiring, and dedicated Windows workflow inclusion.
