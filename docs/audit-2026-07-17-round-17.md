## Material findings fixed

### Delayed Pomodoro completion overcounted focus time

A running Pomodoro stored both `focusSessionStartedAt` and its intended `targetAt`, but completion and interruption accounting used `now - focusSessionStartedAt`. When Chrome delivered an alarm late after device sleep, browser suspension, or a delayed service-worker wake-up, every minute after the configured work deadline was incorrectly added to total focus time. The same overcount could occur when pausing, resetting, or resetting all clocks after the phase was already overdue.

A centralized `pomodoroFocusElapsedMs()` calculation now uses the earlier of `now` and the persisted phase deadline. Normal running/paused behavior is preserved, while delayed processing can no longer inflate completed or interrupted focus duration beyond the actual Pomodoro phase.

### Automatically started work phases were not counted as started sessions

With `autoStartNextPhase` enabled, completion of a break created and scheduled a running work phase, but no `focusSessionsStarted` statistic was recorded. Manually started work phases incremented the counter, while automatic break-to-work transitions silently did not.

Clock completion now reports whether it started a new work phase. The corresponding start statistic is committed exactly once with the transition.

### Clock state and focus statistics were split commits

Interactive start/pause/reset, **Reset all clocks**, and Pomodoro completion committed runtime/revision/alarms first and wrote focus statistics afterward. A statistics storage failure therefore returned an error even though the clock action had already succeeded, left clock state and statistics inconsistent, and made retry semantics ambiguous.

A reusable companion-storage transaction now commits, under the shared `data-write` lock:

- normalized clock runtime;
- related Focus Statistics changes;
- the shared data revision;
- the complete desired durable-alarm set.

The transaction snapshots exact key presence and values. If the focus-stat write, runtime write, revision write, or alarm reconciliation fails, exact prior runtime, Focus Statistics, revision metadata, legacy-key presence, and alarms are restored. Incomplete rollback remains visible as an `AggregateError`.

Non-Pomodoro Timer and Stopwatch actions skip the focus-stat companion entirely, so an unrelated future Focus Statistics schema cannot block, downgrade, or rewrite those clocks.

## Regression coverage

Round 17 executable fixtures verify:

- an alarm delivered ten minutes late records only the configured one-minute work phase;
- delayed completion stores the same deadline-capped duration in Focus Statistics;
- automatic break-to-work transition increments `focusSessionsStarted` exactly once;
- break completion does not increment completed work sessions;
- a forced focus-statistics write rejection restores exact runtime, Focus Statistics, data revision, and alarms;
- reset-all on an overdue Pomodoro reports only time up to the persisted deadline;
- reset-all commits interruption counts and duration with clock state;
- Timer completion succeeds without reading or rewriting an unrelated future Focus Statistics schema;
- generated/static architecture guards reject post-commit clock-stat writes and uncapped Pomodoro duration arithmetic.
