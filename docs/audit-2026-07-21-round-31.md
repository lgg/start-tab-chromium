# Post-Round 30 deep audit — bounded DNR ownership, collision retry, and race-safe Browser Sync frames

## Scope

Independent adversarial review of the exact `master` state produced by PR #107. The pass rechecked the new collision-free DNR allocator, ownership isolation between future Start Tab features, Browser Sync upload commit behavior under competing devices, and the complete regression/CI contract.

## Material findings and fixes

### 1. DNR ownership still matched unrelated same-page redirects

Round 29 introduced an ownership predicate, but it classified every redirect whose URL started with `blocked.html?site=` as a blocklist rule. Another Start Tab feature could legitimately reuse the blocked page with a different query contract or condition and have its rule silently removed during synchronization.

Fix:

- current blocklist rules carry the explicit query marker `owner=start-tab-blocklist-v1`;
- ownership parsing requires the exact extension protocol and host, blocked-page pathname, canonical site, action shape, domain-specific priority, one matching `requestDomains` entry, `main_frame`, and no additional condition fields;
- parser comparison avoids `URL.origin`, which is opaque for `chrome-extension://` URLs and cannot distinguish different extension IDs;
- exact pre-marker rules remain recognized only inside the historical `1…5000` ID range and migrate automatically;
- same-page redirects with another owner, another extension ID, a different priority, or a different rule shape remain foreign and are preserved.

### 2. Collision-free allocation still failed on a concurrent foreign addition

Round 30 allocated around the foreign rule IDs from one snapshot, then performed a second ownership read before the atomic update. If another feature added a rule into the selected ID between those reads, the operation failed even though recalculating immediately would produce a valid free ID.

Fix:

- ownership collisions use a dedicated internal error;
- reconciliation re-reads all dynamic rules and reallocates after that retryable condition;
- retries are bounded to four attempts;
- quota, validation, and unrelated API failures remain non-retryable;
- exact rollback keeps its original IDs and still fails loudly if restoration is genuinely impossible.

### 3. Browser Sync stale cleanup could corrupt a competing upload

Browser Sync previously wrote active chunks and metadata, then removed chunk keys left over from the preceding snapshot. A second device could commit a larger snapshot between those operations; the first device would then delete chunks belonging to the newer snapshot. Its final metadata check detected the conflict, but the remote backup was already incomplete.

Fix:

- every upload builds one complete frame containing metadata plus all 12 canonical chunk slots;
- inactive slots are written as empty strings in the same `chrome.storage.sync.set()` operation;
- metadata `chunks` must exactly equal the number of active chunks supplied to the frame constructor;
- there is no post-commit stale-key removal and no destructive stale rollback;
- checksum, metadata confirmation, item quota, total quota, and unrelated-key preservation remain unchanged.

## Regression coverage

Round 31 executable fixtures verify:

- a foreign redirect using the same blocked page and another owner survives;
- an otherwise exact same-path redirect from another extension ID remains foreign despite opaque URL origins;
- exact in-range legacy blocklist rules migrate to the current ownership marker;
- a high-ID legacy lookalike remains foreign;
- a foreign ID inserted between allocator reads triggers a bounded retry and successful installation at the next free ID;
- the concurrent foreign rule remains unchanged;
- the Browser Sync frame always contains all 12 chunk slots;
- inactive slots are cleared inside that frame;
- mismatched metadata/frame chunk counts are rejected.

Static contracts preserve the exact ownership parser, extension-host comparison, bounded legacy migration, bounded collision retry, complete sync-frame constructor, absence of stale-key removal, package/CI wiring, and self-hosted regression enumeration.

The exact final PR head must pass every historical regression through Round 31, all release-validation scopes, the self-hosted Windows CI contract, strict TypeScript typecheck, full extension build, blocker-only build, explicit Google-enabled build, generated-output validation, bounded cleanup, and every action post-step. Temporary patch infrastructure must be absent from the final diff.
